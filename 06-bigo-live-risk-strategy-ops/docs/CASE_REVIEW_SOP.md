# Case 复盘与标准沉淀 SOP

## 1. Bad case 闭环

```text
发现
  ↓
还原链路
  ↓
定位根因
  ↓
补策略
  ↓
灰度验证
  ↓
记录 SOP 和案例库
```

## 2. 复盘字段

| 字段 | 说明 |
|---|---|
| case_id | 案例编号 |
| risk_type | 风险类型 |
| source | 发现来源：举报、巡查、模型、业务反馈 |
| actor | 风险主体：账号、设备、IP、收付款账号、支付类别、公会 |
| behavior_chain | 行为链路 |
| harm_scale | 危害规模：触达人数、引流人数、金额、举报量 |
| current_strategy | 当前命中或漏过的策略 |
| root_cause | 根因：规则缺失、特征缺失、阈值问题、审核不一致、对抗变体 |
| action | 补充动作：加规则、调阈值、加名单、补 SOP |
| review_result | 灰度和回看结果 |

## 3. 根因分类

| 根因 | 说明 | 处理方向 |
|---|---|---|
| 规则缺失 | 完全没有覆盖该风险 | 新增规则或图谱策略 |
| 特征缺失 | 数据里有信号但没抽出来 | 新增特征表或指标 |
| 阈值问题 | 阈值过松或过严 | 离线回放后调阈值 |
| SOP 不一致 | 人审判定标准不统一 | 更新审核指南和样例 |
| 黑产对抗 | 对方更换话术、设备、路径 | 增加变体识别和关联图谱 |

## 4. 案例库沉淀

案例库至少包含：

- 典型截图或文本样例的脱敏描述。
- 风险链路图。
- 命中 / 未命中规则。
- 复盘结论。
- 后续策略动作。
- 是否已纳入 SOP。

## 5. 面试表达

> 我会把每个 bad case 按“发现、还原、根因、补策略、验证、沉淀”的闭环处理，而不是只停留在补一条规则。这样可以把一次问题变成可复用的风险标签、判定 SOP 和案例库。
