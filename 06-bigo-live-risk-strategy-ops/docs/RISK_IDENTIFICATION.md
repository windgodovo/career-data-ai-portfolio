# 风险识别与问题定义

## 1. 为什么先定义风险

直播风控不能从“看到了一个违规内容”开始，而要先明确：平台到底要防什么、谁在做、行为链路是什么、危害规模多大、现有策略为什么没挡住。

## 2. 风险标签体系

| 一级风险 | 二级风险 | 典型表现 | 处置优先级 |
|---|---|---|---|
| 涉政 | 敏感议题、组织动员、恶意传播 | 直播间话题聚集、房间标题暗示 | P0/P1 |
| 色情 | 裸露暗示、色情交易、站外引流 | 低俗动作、私信诱导、外部联系方式 | P0/P1 |
| 暴力 | 暴力展示、威胁恐吓、极端内容 | 画面或语音中出现暴力威胁 | P0/P1 |
| 诈骗 | 假福利、冒充身份、诱导转账 | 诱导充值、外部支付、骗取礼物 | P0/P1 |
| 诱导 | 站外 App、群组、联系方式引流 | 口播、文字变体、头像/昵称暗示 | P1/P2 |
| 黑灰产 | 批量养号、团伙矩阵、设备簇 | 同设备/IP/支付类别批量行为 | P1/P2 |

## 3. 风险问题定义模板

```text
风险名称：
发现来源：举报 / 巡查 / 模型命中 / 业务反馈 / 舆情
谁：账号、主播、设备、IP、收付款账号、支付类别、公会
做了什么：开播、发言、私信、引导、收礼、跳转外部平台
行为链路：账号注册 -> 进入房间 -> 互动 -> 引导 -> 转化
造成危害：触达用户数、被引流用户数、交易金额、举报量、投诉量
现有策略：当前命中哪些规则，哪些没有命中
初步判断：漏过 / 误杀 / 策略空白 / 审核标准不一致
```

## 4. 示例问题定义

```text
风险名称：语音房新号色情引流
发现来源：用户举报 + 人工巡查
谁：注册 24 小时内的新账号，集中在同设备簇和同 IP 段
做了什么：进入语音房后重复发送变体联系方式，引导用户站外沟通
造成危害：日均触达 2 万用户，疑似引流成功 300-500 人
现有策略：关键词规则命中标准表达，但漏掉变体表达和图片昵称暗示
初步判断：策略漏过 + 特征缺失
```

## 5. 在哪里看高风险 / 低风险事件

当前项目里，“高风险 / 中风险 / 低风险”不是单独写在原始数据中，而是由 SQL 和 Python 根据举报、处罚、设备/IP 聚集、私信行为、支付金额和审核结果计算出来。

### 5.1 先看 SQL 输出表

运行完整链路后：

```bash
python3 scripts/run_pipeline.py
```

可以在 `data/output/` 下查看这些结果表：

| 文件 | 看什么 | 适合判断 |
|---|---|---|
| `risk_strategy_candidate.csv` | 用户级策略候选，包含私信数、举报数、处罚数、设备/IP 聚集、支付金额 | 哪些用户应该进入复审、限流或观察名单 |
| `suspect_cluster_result.csv` | 设备、IP、支付类别维度的疑似团伙簇 | 哪些设备/IP/支付类别聚集了多个风险用户 |
| `bad_case_review.csv` | 审核 case 和账号风险特征合并后的复盘表 | 哪些 case 是团伙、私信滥用、可能误伤或普通单 case |
| `risk_metric_daily.csv` | 每天各风险类型的处罚量 | 哪天、哪类风险规模变大 |
| `payment_flow_risk.csv` | 支付类别下的订单数、金额和退款率 | 哪些支付类别 / 支付场景可能异常 |

### 5.2 再看 Python 分析输出

运行分析脚本：

```bash
python3 python/risk_insight_workflow.py
```

脚本会把结果导出到：

```text
python/analysis_output/
```

其中最直接的风险事件文件是：

| 文件 | 含义 |
|---|---|
| `high_risk_events.csv` | 高风险事件，当前主要是设备/IP/支付链路聚集且处罚、举报都较集中的 case |
| `medium_risk_events.csv` | 中风险事件，有举报、私信或处罚信号，但团伙性不如高风险明显 |
| `low_risk_events.csv` | 低风险事件，正常 case 或可能误伤 case，通常不应直接强处置 |
| `risk_event_ranking.csv` | 所有 case 的风险排序总表，包含 `risk_level` 和 `risk_score` |
| `ranked_suspect_clusters.csv` | 疑似团伙簇排序，适合看设备/IP/支付类别聚集风险 |
| `metric_spikes.csv` | 风险指标突增检测结果 |
| `bad_case_summary.csv` | 对排名最高团伙线索生成的复盘摘要 |

### 5.3 当前风险等级口径

Python 脚本里的 `risk_event_ranking.csv` 会按以下规则给 case 分层：

| 风险等级 | 判断逻辑 | 处理建议 |
|---|---|---|
| high | `root_cause = cluster_abuse`，通常意味着同设备/IP/支付链路下多个用户集中被举报和处罚 | 优先复审，必要时联动设备/IP/支付类别限流 |
| medium | 有举报、私信滥用或处罚信号，但团伙聚集不强 | 进入观察名单或人工抽检 |
| low | 审核为正常，或申诉推翻，或信号较弱 | 避免直接强处罚，可用于误杀复盘 |

所以面试里可以这样讲：

```text
我不会直接从原始表里判断高低风险，而是先通过 SQL 汇总用户、房间、团伙簇和支付链路特征，再用 Python 把 bad case 按 root_cause、举报数、私信数、设备聚集和支付金额做风险分层。最终高风险、中风险、低风险事件会落到 python/analysis_output/ 下面，业务同学可以直接打开 CSV 做复审和策略讨论。
```
