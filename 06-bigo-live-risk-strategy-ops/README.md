# 大型直播平台 BIGO LIVE 直播风控策略运营项目

这是一个面向大型直播 / 语音房平台的风控策略运营作品集项目。项目以 BIGO LIVE 这类跨地区、强互动、强实时的平台为参考场景，围绕直播间、语音房、账号、设备、支付、举报、审核和封禁数据，设计一套可落地的风险识别、数据洞察、策略迭代、指标监控、Case 复盘和跨部门协同方法。

项目定位不是“内容审核员”，也不是单纯“跑数分析师”，而是更偏**直播风控策略运营 / 安全策略分析 / Trust & Safety Analytics**：能把模糊的举报和巡查现象抽象成风险问题，再用 SQL + Python 找规律、提策略、看指标、做复盘。

## 1. 项目目标

- 建立直播 / 语音房场景下的风险标签体系。
- 将举报、审核、封禁、账号行为、设备/IP、支付和消息数据转化为可分析的风险特征。
- 识别团伙、养号、黑灰产引流、内容模板复用、异常互动和异常支付链路。
- 把数据洞察转成可上线的策略规则、阈值、名单和关联图谱维度。
- 监控漏过率、误杀率、处置量、申诉率、复审推翻率等核心指标。
- 沉淀 Bad case 复盘机制、判定 SOP 和典型案例库。

## 2. 业务背景

大型直播平台的风险不是单点行为，而是账号、内容、互动、支付和外部引流链路共同构成的问题。例如：

```text
新账号批量注册
  ↓
短时间进入语音房 / 直播间
  ↓
使用相似话术或头像昵称
  ↓
诱导用户私聊 / 加外部联系方式 / 充值打赏
  ↓
诈骗、色情引流、黑灰产转化
```

因此风控策略运营要回答的问题不是“有没有违规”，而是：

- 谁在做：账号、主播、公会、设备、IP、收付款账号和支付类别。
- 做了什么：开播、发言、私信、引导、收礼、跳转外部平台。
- 怎么做的：话术模板、时间规律、设备簇、支付簇、账号链路。
- 造成什么危害：影响用户数、转化金额、举报量、外部引流规模。
- 现有策略漏在哪：规则缺失、阈值过松、特征滞后、审核 SOP 不一致。

## 3. 核心应用链路

```text
订单 / 消息 / 举报 / 审核 / 封禁 / 账号 / 设备 / IP / 支付
        ↓
风险识别与问题定义
        ↓
SQL + Python 特征提取与洞察分析
        ↓
策略规则 / 阈值 / 黑白名单 / 关联图谱迭代
        ↓
灰度上线 / 指标回看 / 放量 / 复盘
        ↓
风险标签体系 / SOP / 案例库沉淀
```

## 4. 六个重点模块

### 4.1 风险识别 & 问题定义

先把“坏什么”说清楚。直播 / 语音房常见风险包括：

| 风险类型 | 典型表现 | 需要抽象成的问题 |
|---|---|---|
| 涉政 | 敏感议题煽动、组织动员、恶意传播 | 内容风险 + 传播规模 |
| 色情 | 暗示性内容、色情引流、诱导私聊 | 主播行为链 + 用户转化链 |
| 暴力 | 暴力展示、威胁恐吓、极端内容 | 内容等级 + 处置时效 |
| 诈骗 | 冒充身份、诱导转账、假福利 | 账号链路 + 支付链路 |
| 诱导 | 诱导站外、诱导充值、诱导加群 | 内容模板 + 外链转化 |
| 黑灰产引流 | 批量账号、设备簇、话术复用 | 团伙识别 + 规模控制 |

问题定义模板：

```text
谁：账号 / 主播 / 设备 / IP / 收付款账号 / 支付类别 / 公会
做了什么：开播、发言、私信、引导、收礼、跳转外部平台
造成什么危害：触达用户数、被引流用户数、交易金额、举报量、投诉量
现有策略状态：已拦截、漏过、误杀、待复审
```

### 4.2 数据提取 & 洞察

最核心能力是 SQL + Python：不是简单跑数，而是从多张行为表中提炼风险结论。

可提取的数据源：

- 账号数据：注册时间、国家/地区、认证状态、设备、IP、登录历史。
- 直播数据：开播时长、房间类型、观众数、互动数、违规命中记录。
- 消息数据：公屏、私信、语音房文本转写、关键词、URL、联系方式。
- 举报数据：举报原因、举报人、被举报对象、举报时间、命中结果。
- 审核封禁：审核结果、处罚类型、封禁时长、复审结果、申诉结果。
- 订单支付：充值、打赏、退款、异常支付类别、礼物流向。

洞察方向：

- 团伙特征：同设备、同 IP、同支付类别 / 支付场景、同注册时间段、同话术模板。
- 时间簇：夜间集中作案、活动期突增、某时区集中爆发。
- 内容模板复用：昵称、头像、简介、话术、链接、外部联系方式相似。
- 引流跳板：直播间 → 私信 → 外部群 / 外部 App / 支付链路。
- 策略缺口：漏在哪类国家、语种、房间类型、账号年龄、设备簇。

### 4.3 策略 / 规则迭代

把洞察变成可执行策略：

| 策略类型 | 示例 |
|---|---|
| 阈值规则 | 新账号 24 小时内高频私信 + 多次被举报，进入人工复审 |
| 黑名单 | 命中高危设备 / IP / 收付款账号 / 支付类别，直接限制开播或私信 |
| 白名单 | 历史稳定主播、官方活动账号降低误杀风险 |
| 关联图谱 | 账号-设备-IP-支付-房间关系图识别团伙 |
| 分层处置 | 警告、限流、禁言、下播、短封、长封、人工复审 |
| 灰度策略 | 小流量国家 / 语种 / 房间类型先试点，再全量放开 |

核心权衡：

- 提高召回：减少风险漏过。
- 控制误杀：避免正常主播、正常互动被中断。
- 控制体验损失：低风险先警告或复审，高风险才硬拦。

### 4.4 指标 & 监控

策略是否有效，要靠指标证明。

| 指标 | 含义 |
|---|---|
| 漏过率 | 违规后未被策略及时命中的比例 |
| 误杀率 | 正常内容 / 正常账号被错误处置的比例 |
| 处置量 | 策略触发的警告、禁言、下播、封禁数量 |
| 申诉率 | 被处置对象发起申诉的比例 |
| 复审推翻率 | 人工复审后撤销原处罚的比例 |
| 风险规模趋势 | 按日、周、活动期观察风险量级变化 |
| 策略命中率 | 某条策略命中后最终被确认违规的比例 |
| 人审压力 | 进入人工复审的 case 数和平均处理时长 |

报警思路：

- 今日必须介入：风险量突增、误杀率突增、复审推翻率异常、活动期黑产爆发。
- 次日优化：规则边界不清、某语种误判偏高、某类模板漏过增加。

### 4.5 Case 复盘 & 标准沉淀

Bad case 闭环：

```text
发现问题
  ↓
还原行为链路
  ↓
定位根因
  ↓
补策略 / 调阈值 / 加特征
  ↓
灰度回看
  ↓
记录到案例库和 SOP
```

根因分类：

- 规则缺失：现有策略没有覆盖这类风险。
- 特征缺失：数据里有信号，但没有被提取成特征。
- 阈值问题：规则存在但阈值过松或过严。
- 审核不一致：人工判定 SOP 不清晰。
- 黑产对抗：对方换话术、换设备、换国家、换支付路径。

### 4.6 跨部门协同

风控不是躲在数据后面。策略运营需要和多方协同：

| 协作对象 | 需要对齐的问题 |
|---|---|
| 审核运营 | 哪些风险必须硬拦，哪些进入复审，哪些只警告 |
| 安全运营 | 团伙、设备、IP、支付链路是否需要联动处置 |
| 产品 | 是否需要入口限制、私信限制、开播门槛或用户教育 |
| 直播业务 | 如何减少对正常主播和活动流量的影响 |
| 数据 / 算法 | 哪些特征可稳定产出，哪些可训练模型增强召回 |

关键能力：把数据结论翻译成业务语言。例如不是说“某规则 precision 下降”，而是说：

> 最近东南亚语音房里，新注册 1 天内账号通过重复话术引导用户站外联系，现有关键词规则漏掉了变体表达，预计日均漏过 300-500 个风险互动。建议先对新号高频私信 + 被举报组合做灰度限流，同时补充变体词表和人工复审 SOP。

## 6. 运行方式（可直接跑）

安装依赖并执行全链路：

```bash
cd "06-bigo-live-risk-strategy-ops"
python3 -m pip install -r requirements.txt
python3 scripts/run_pipeline.py
```

运行后会在 `data/output/` 生成账号风险、房间风险、话术复用、团伙簇、行为链、支付链路、策略候选、指标监控和 Bad case 复盘等 10 张结果表，在 `data/warehouse/` 生成 DuckDB 文件。

逐层检查每步 SQL 和中间表：

```bash
python3 scripts/inspect_sql_results.py
```

这个脚本会把每张中间表导出到：

```text
scripts/csv_preview/
```

把导出的结果继续做洞察分析：

```bash
python3 python/risk_insight_workflow.py
```
